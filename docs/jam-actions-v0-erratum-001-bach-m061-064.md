# jam-actions-v0 — Erratum 001: `bach-prelude-c-major-bwv846:m061-064` window exceeds the song

**Status:** fixed in the working set 2026-07-11 (revision r001); ships with the next public package cut. The sealed published package **v0.4.3 (Zenodo DOI 10.5281/zenodo.20279919) is unchanged** — its checksums and release gate were re-verified after the fix (see §Verification).

## The finding

The published record `bach-prelude-c-major-bwv846:m061-064:piano:mcp-session:v1` carries a frozen `play_song(id: bach-prelude-c-major-bwv846, startMeasure: 61, endMeasure: 64, mode: loop)` call that the live MCP server rejects:

> `endMeasure (64) exceeds "Prelude in C Major, BWV 846 (Well-Tempered Clavier)"'s length. Valid range: 1-62.`

Found 2026-07-11 by the finetune-arc-v1 execution-verification gate — **G6a** executed every unique frozen jam tool call in the training corpus against the real server over MCP stdio and this was the single failure out of 206 unique calls ([P1v1-gate-report.json](../experiments/finetune-arc-v1/data/P1v1-gate-report.json)). Disposition recorded ex-ante as amendment **A1-v1** in [experiments/finetune-arc-v1/P0-LOCK.md](../experiments/finetune-arc-v1/P0-LOCK.md): gate verdict `PASS_WITH_FINDINGS`, fix filed to the dataset backlog for a future revision. This document is that fix.

## Ground truth (established 2026-07-11)

- The source MIDI (`songs/library/classical/bach-prelude-c-major-bwv846.mid`, piano-midi.de, Bernd Krüger, unchanged in git since import) is **exactly 62 measures**: the file is "Praeludium und Fuge 1 in C-Dur BWV 846" — **prelude mm. 1–35 + fugue mm. 36–62** concatenated (track names confirm; last note-off tick 119040 = 62 × 1920 exactly).
- The server registry has said 62 measures since the MIDI-first architecture (2026-02) — including at dataset build time. This was **not** a server regression: re-running both the pre- and post-2026-07-09 note-resolution logic on the MIDI yields 62 either way.
- The record's own payload was always honest: its MIDI sidecar and REMI tokens contain **only bars 61–62** (41 events, `Bar_61`/`Bar_62`). Only the *labels* and *synthesized trace* claimed four measures.
- Root cause: the Slice-9b corpus builder's hand-authored Bach song spec assumed "~70 measures" and authored a final phrase pair at mm. 61–64. The phrase slicer is silent on out-of-range windows (documented: "empty result is not an error"), the schema validates no window against real song length, and the trace's tool results are synthesized from the spec — turn 3 claimed `measures: 4` and turn 5 claimed `playback_started: true` without ever executing the calls. v0's schema-only validation (G2) could not see any of this; execution verification (G6a) could.
- Blast radius audited across all 145 working-set records: **this is the only out-of-bounds window** in the corpus.

## Decision

**Correct the record for the next dataset revision. The server and the song registry are already right.** Alternatives rejected:

1. **Extend the server's registry entry** — impossible. There is no bar 63–64 music; the piece ends at m62. The registry reflects MIDI tick truth, which the dataset's own design doctrine treats as canonical.
2. **Make `play_song` clamp like `view_piano_roll`** — rejected. The strict bounds check was added deliberately (2026-07-09, F-c969321e) after out-of-range loops crashed background playback; silently looping different measures than requested is worse UX for a teaching platform than a clear error naming the valid range. The asymmetry is intentional: a read-only render may clamp to what exists; a side-effectful playback action must not reinterpret the request. Clamping would also have hidden this authoring error forever — the gate only caught it because the server refused.

## Known residuals (filed separately, not part of r001)

- **Annotation prose vs. actual music:** the Bach records from m033 onward describe prelude material ("dominant pedal", "final arpeggios"), but measures 36–62 of this MIDI are the **fugue**. The r001 record keeps its prose (numbers corrected only); a content-accuracy pass over the Bach annotations is a separate, director-scoped revision — it changes training text, not tool-call validity. **→ Resolved 2026-07-11 by revision r002 (tier AB — all 16 Bach records, ground-truth-verified): [erratum-002](jam-actions-v0-erratum-002-bach-annotation-prose.md).**
- **Synthesized trace semantics:** turn-3 `measures` equals the window size by construction. With the builder guard in place, window size always equals rendered size, so the value is now honest — but traces remain synthesized, not captured; execution verification (the G6a pattern) is the compensating control and is recommended as a standing packaging gate for any future version.
