## 4. Audit Phase 2 — MCQ gold bugs

### 4a. Pathétique m017-020 off-by-one (CONFIRMED + FIXED)

**Reproduction:**
- Record `pathetique-mvt2-m017-020.json`, measure 19, right-hand events:
  - beat 0.6604 → MIDI 70 → A#4
  - beat 1.6729 → MIDI 63 → D#4
  - beat 3.4854 → MIDI 63 → D#4
- Slice 18 MCQ output: question stem said "beat 1.6604", gold option A#4 (correct identity), but A#4 lives at beat 0.6604.

**Root cause:** `annotation-grounding.ts` line 756 (pre-Slice-18.5) computed:
```ts
const beatDisplay = anchor.beat + 1;
```
The comment said "1-indexed display for readability." The bug: the inspector tool `get_pitch_at` consumes 0-indexed beats (matches `timed_events.beat`), and the model, given a question text saying "beat 1.6604", queries the tool with 1.6604. The tool's `BEAT_EPSILON=0.1` tolerance lands on the next nearest event (D#4 at 1.6729), not the gold A#4 (which is one full beat earlier at 0.6604).

This was a **class of bug**, not a single record's idiosyncrasy. EVERY annotation_grounding question in EVERY record had a beat display 1.0 higher than the actual event. On Pathétique m017 the gap landed across an event boundary; on most other records the gap landed in a quiet gap or at end-of-measure, so the tool returned null (the slice 18 17.5% "null-tool" sub-class). Both failure modes share the same root cause.

**Fix:** remove the `+1` shift; display the actual stored beat.
```ts
// Slice 18.5 fix in annotation-grounding.ts
const beatLabel = `beat ${anchor.beat}`;
```

**Regression test (new):** `describe("Slice 18.5 regression — pathetique-mvt2:m017-020 off-by-one", ...)` in `annotation-grounding.test.ts`. Two tests:
1. Specific Pathétique m017 case: A#4 lives at beat 0.6604, the question text contains "beat 0.6604" exactly, `midiClaim.beat` matches, and `q.options[q.correctOptionIndex] === noteName(midiClaim.note)`.
2. Defense-in-depth: across EVERY public record, the displayed beat in the annotation_grounding question text matches `midiClaim.beat` exactly. Locks the off-by-one class closed across the corpus.

**Test revision:** the existing "question text specifies measure, hand, and beat" test still passes (it only required `/beat/` in the text). A new test added next to it pins the load-bearing invariant: `displayedBeat === midiClaim.beat`.

### 4b. Systematic scan — other beat/quantization bugs (NONE FOUND)

Audited all beat-using generators for similar off-by-one / nearest-event / rounding issues:

- **`generateRhythmOnsetQuestion`** (lines 558–572): uses `e.beat === 0` / `e.beat === 1.0` / `e.beat < 0.5` heuristic. The matching inspector tool `count_beat_1_onsets` uses the SAME heuristic line-for-line. No drift, no off-by-one. OK.
- **`generatePitchClassCountQuestion`** (lines 399–434): pure pitch-class count; no beat arithmetic. OK.
- **`generateHandRegisterQuestion`** (lines 452–518): counts events by `e.hand` field. No beat arithmetic. OK.
- **`generateAnnotationGroundingQuestion`** (lines 680–777): the ONE bug above. After the fix, the displayed beat matches `midiClaim.beat` matches the tool's 0-indexed query convention end-to-end.

Single class of bug fixed. No additional MCQ generator changes.

## 5. The new tool — `count_notes_with_pitch_class`

**Signature:**
```ts
function count_notes_with_pitch_class(
  recordOrEvents: E3Record | TimedEvent[],
  pitchClass: unknown,
): { pitch_class: string | null; count: number; error?: string };
```

**Input convention:** pitch class as a STRING (chosen as the canonical input shape because the MCQ generator emits pitch class NAMES in question text: "G", "C#", "A", etc.). Accepts:
- canonical sharp names: "C", "C#", "D", "D#" … "B"
- flat names: "Db", "Eb", "Bb", "Ab" (case-insensitive)
- unicode flats: "C♭", "B♭"
- octave-prefixed notes: "C4" → strips to "C"
- integer 0–11 as a string: "7" → "G"

**Output:** `{ pitch_class: <canonical sharp form>, count: <integer>, error?: <string when input unrecognized> }`. Errors return count=0 with `pitch_class: null` rather than throwing — matches the existing "tools must handle malformed args gracefully" rule.

**JSON schema:** `COUNT_NOTES_WITH_PITCH_CLASS_SCHEMA` exported alongside the function; registered in `INSPECTOR_TOOLS` at position 4 (between `count_distinct_pitch_classes` and `count_beat_1_onsets`). Brings the inspector surface from 8 → 9 tools.

**Test coverage** (10 new tests in `midi-inspector.test.ts`):
- counts G in Bach m1-4 against hand-computed expectation
- counts D# in pathetique against direct event filter
- normalizes Eb → D#
- normalizes B♭ (unicode) → A#
- strips trailing octave ("C4" → "C")
- accepts "7" → "G"
- case-insensitive on letter ("g" → "G")
- returns count=0 with empty events
- deterministic across multiple calls
- routes via tool registry (`findInspectorTool("count_notes_with_pitch_class")`)
- handles malformed args without throwing (missing field, integer overflow, null)

**Tool catalog updates:**
- `midi-inspector.ts` header catalog: "8 tools" → "9 tools" (count_notes_with_pitch_class added)
- `midi-inspector.test.ts`: "exposes 8 tools" → "exposes 9 tools"; "inspectorToolSchemas() returns 8" → "returns 9". Comments on each REVISED test note `Slice 18.5: count_notes_with_pitch_class joined the registry.`

**`annotation-grounding-tool.ts` (Slice 17 multi-turn loop) — byte-identical.** The runner already consumes `inspectorToolSchemas()` and `findInspectorTool()` dynamically, so adding a 9th tool flows through with zero source change in the runner.

- schumann m045-048: model misinterprets `get_pitch_at` results against the options. Pre-existing failure pattern.

These residuals are honest model behavior. They cannot be made smaller by tool-surface or MCQ-gold fixes; they would require either a stronger base model (hermes3:8b cross-comparison is plausible) or post-tool-call reasoning scaffolding (chain-of-thought or self-check loops, both out of scope for this slice).
